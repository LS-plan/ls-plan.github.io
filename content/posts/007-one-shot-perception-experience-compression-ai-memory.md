上一篇我写到一句话：

> 把一次操作过程整理成博客，不只是记录流水账，它更像是在做一次“经验压缩”。这背后是一种工程师非常重要的思维：把一次性经验变成可迁移模型。

写完以后，我想到一个很像的现象：有些黑白二值图，第一次看是一团奇怪的斑块；但只要看过原图，知道里面是一只狗，再回头看那张黑白图，就再也“看不回去了”。你会稳定地看到狗。

这类现象常见于 Mooney images 或类似的退化图像研究。它很迷人，因为它说明人的感知不是单纯从像素往上堆特征，而是会被一次关键经验迅速改写：原来看不出来的东西，一旦被解释过，就会变成一种新的可复现知觉。

这和我说的“经验压缩”有没有关系？

有关系，但不是同一件事。

它们都涉及一件核心事情：

> 一次经验如果被组织得足够好，就不只是被保存下来，而是会改变后续解释世界的方式。

---

## 一、什么是“单次感知能力”

我这里说的“单次感知能力”，更接近认知科学和计算机视觉里的 **one-shot perceptual learning**。

典型过程是：

1. 你先看到一张退化图像，比如黑白二值化、遮挡、轮廓模糊的图。
2. 你看不出里面是什么。
3. 别人给你看原图，或者告诉你答案。
4. 你再看退化图像，就能稳定看出目标。
5. 以后再看这张图，很难回到“看不出来”的状态。

这不是简单记住一张图片，而是获得了一个新的解释模板。

认知上，它像是把图像里的零散斑块重新绑定成一个整体对象。原先的输入没有变，变的是你的先验、注意力和特征组织方式。

也就是说：

```text
同一张图 + 新先验 = 新感知
```

这件事很像工程复盘：

```text
同一类问题 + 新模型 = 新判断
```

比如你第一次看到 `nobodyls.cn`、GitHub Pages、Caddy、Docker、`localhost:8092` 混在一起，会觉得它们是一团线。但一旦你把它压缩成：

```text
DNS -> 入口层 -> 服务层 -> 路径兼容
```

下次再遇到类似部署问题，你就不会从命令开始乱试，而是先问：流量到底有没有到这台机器？

这就是经验压缩的价值。

---

## 二、它和“经验压缩”的相同点

单次感知和经验压缩都不是“保存更多细节”，而是“形成更好的索引和解释结构”。

### 1. 都依赖先验

人能从一张退化图像里看到狗，不是因为图像突然多了像素，而是因为你已经有关于狗、轮廓、姿态、头身关系的先验。

工程经验也是一样。

一次 Caddy 配置经历之所以能迁移，是因为你已经积累了一些底层概念：

- 域名解析
- 端口监听
- HTTP Host
- 路径路由
- 反向代理
- 静态资源路径

经验压缩不是把这些概念重学一遍，而是把它们重新组织成一个更容易调用的判断框架。

### 2. 都会改变后续识别

看过原图以后，你再次看到黑白图，会更快、更稳定地识别目标。

写过复盘以后，你再次看到类似问题，也会更快定位层级：

```text
这不是代码问题，是 DNS 没切。
这不是服务挂了，是路径前缀没剥。
这不是 Caddy 失效，是 GitHub Pages 的 baseURL 写死了根路径。
```

这就是一次经验对后续判断的改写。

### 3. 都不是完整泛化

你提到一个关键点：如果图像旋转、变形、换角度，这种效果会大打折扣。

这说明单次感知很强，但它不是万能抽象。它常常绑定在具体视角、具体结构、具体提示上。

经验压缩也一样。

如果我只记住：

```text
Caddy 加 handle_path 就好了
```

那换到 Nginx、Cloudflare Tunnel、Kubernetes Ingress 时就可能失效。

但如果我压缩成：

```text
先确认入口层，再确认路径是否需要重写
```

它的迁移性就强很多。

所以经验压缩的质量，取决于你压缩出来的是“表面命令”，还是“可迁移结构”。

---

## 三、它们的不同点

虽然相似，但单次感知和工程经验压缩不是同一层机制。

### 单次感知偏感知绑定

它解决的是：

> 这团输入到底应该被组织成什么对象？

它更接近视觉系统里的先验激活、轮廓整合、对象识别。

### 经验压缩偏问题求解

它解决的是：

> 下次遇到类似问题，我应该按什么顺序判断？

它不只是识别，还包括：

- 分层
- 因果解释
- 操作顺序
- 风险控制
- 验证方法
- 失败回滚

如果用一句话区分：

> 单次感知让你“看见”；经验压缩让你“会处理”。

## 四、现在的 AI 工具为什么很难真正拥有这种经验

现在的 Codex、Claude Code、各类 Agent 工具，已经能在一个会话里表现得很像“学会了”。

但大部分时候，它们只是：

```text
当前上下文里有这段经验 -> 模型注意到了 -> 临时利用
```

这不是长期学习。

模型权重没有因为这次会话改变。下次开新会话，如果你不给它上下文，它通常不会自动记得：

- 你的服务器 Caddy 配置结构
- 你博客的 Hugo 目录
- 你偏好的文章风格
- 你上次踩过的路径代理坑
- 你希望“知识利用流动”成为个人门户主线

这就是上下文窗口和长期记忆之间的差别。

上下文像工作台。

长期记忆像工具柜。

> 如果你对这个“工作台/工具柜”的类比感兴趣，我在下一篇继续把它展开成“内存/存储/文件系统/调度器”的 AgentOS 视角：[[008] 从内存与存储看 AgentOS：AI 经验如何被保存、调度和复用](/posts/008-agentos-memory-storage-rag-experience/)。

工作台再大，也不等于工具会被整理进柜子。

## 五、RAG 能不能解决这个问题

RAG 的基本思路是：

```text
用户问题 -> 检索相关文档 -> 把文档塞进上下文 -> 让模型回答
```

它很有用，但它本身不等于经验压缩。

因为很多 RAG 知识库只是“文本仓库”：

- 存了全文
- 切了 chunk
- 做了向量索引
- 问的时候召回几段

这能解决“找得到材料”，但不一定解决“复用得好”。

如果把一整段会话直接丢进知识库，后续检索可能会出现几个问题：

1. 噪声太多：探索过程、失败尝试、临时判断混在一起。
2. 意图不清：不知道这段经验适用于什么场景。
3. 粒度不稳：一次会话里可能有十几个不同问题。
4. 缺少验证：不知道最后哪些结论真的被测试过。
5. 难以执行：召回的是聊天记录，不是操作手册。

所以真正有价值的不是“把会话存进 RAG”，而是把会话压缩成经验对象。

## 六、一个会话如何压缩成经验

我觉得可以把一次会话沉淀成一种“经验卡片”。

它不是总结，也不是日志，而是一个可复用的知识单元。

一个好的经验卡片至少包含这些字段：

```yaml
title: "Caddy 子路径代理导致分享链接丢到根路径"
applies_when:
  - "应用原本部署在根路径"
  - "现在挂到 /image/ 这类子路径"
  - "前端或后端返回 /share/:id 这种绝对路径"
symptoms:
  - "页面主体可访问"
  - "分享链接跳到 https://domain/share/id"
  - "根域名返回 404"
root_cause:
  - "服务端返回绝对路径"
  - "前端用 new URL(data.path, window.location.href) 拼接"
  - "绝对路径绕开了 /image/ 路由"
fix:
  - "服务端返回 ./share/:id"
  - "Caddy 增加 /share/* 兼容路由"
verification:
  - "POST /image/api/share 返回 ./share/:id"
  - "GET /image/share/:id 返回 200"
  - "GET /share/:id 兼容返回 200"
risk:
  - "分享内容存在内存中，容器重启后旧链接失效"
```

这样的卡片比原始聊天记录有用得多。

因为它已经把一次过程压缩成：

```text
触发条件 -> 症状 -> 根因 -> 修复 -> 验证 -> 风险
```

下一次 AI 工具检索到它，不需要重新读完整对话，就能直接复用判断模型。

## 七、这和单次感知的类比在哪里

如果把原始会话比作一张复杂图片，那么经验卡片就是“看过原图之后形成的对象模板”。

没有经验卡片时，AI 工具看到问题，可能只能看到一堆碎片：

```text
Caddy
share
404
路径
image-web
new URL
```

有经验卡片后，它会更容易看到整体：

```text
这是子路径部署下的绝对路径泄漏问题
```

这就很像人看到 Mooney 图像里的狗。

不过要注意一个区别：

人类的单次感知可能把新的先验写进了大脑系统；而现在大多数 AI 工具不会自动改权重。它们的“记住”通常是外部化的：

- 存在文档里
- 存在向量库里
- 存在项目规则里
- 存在 agent memory 里
- 存在下一次 prompt 的检索上下文里

所以 AI 的经验复用更像：

```text
外部记忆系统 + 检索 + 上下文注入 + 模型推理
```

而不是：

```text
模型本体真的学会了这件事
```

## 八、RAG、长期记忆和经验压缩的分工

我会这样分：

| 层级 | 作用 | 类比 |
|------|------|------|
| 原始会话 | 保留完整证据 | 录像 |
| 经验卡片 | 提炼可复用判断 | 剪辑后的教学片段 |
| RAG/知识库 | 检索相关经验 | 图书馆目录 |
| Agent 规则 | 固化行为偏好 | 工作守则 |
| 项目文档 | 记录本项目事实 | 施工图纸 |
| 微调/训练 | 改变模型参数 | 长期技能内化 |

这里最容易误解的是 RAG。

RAG 很像图书馆目录和资料柜，它能把相关材料拿出来。但材料是否“可用”，取决于你之前有没有把经验整理成合适的形态。

如果知识库里存的是：

```text
今天我和 AI 聊了 20000 字，最后修好了。
```

下次召回也很痛苦。

如果知识库里存的是：

```text
子路径部署检查清单：绝对路径、baseURL、API 根路径、分享链接、静态资源、回调 URL。
```

那它才像真正的经验。

## 九、Claude Code / Codex 这种工具应该怎么接入经验

我理想中的工作流是这样的：

### 1. 会话结束时生成经验候选

不是每个会话都值得沉淀。

值得沉淀的通常有这些特征：

- 解决了真实问题
- 中间有误判或关键发现
- 涉及某个项目的长期约定
- 后续大概率会复用
- 有明确验证结果

### 2. 把经验写成结构化文档

例如放在项目里：

```text
.agents/memory/
  caddy-subpath-share-link.md
  hugo-post-publish-checklist.md
  image-web-deploy-contract.md
```

或者放在全局知识库：

```text
knowledge/engineering/
  reverse-proxy-subpath.md
  ai-session-experience-compression.md
```

### 3. 下次任务开始前检索

当你问：

```text
image-web 分享链接在 nobodyls.cn/image/ 下又 404 了
```

Agent 应该能检索到：

- 子路径部署经验卡
- image-web 部署契约
- Caddy 路由片段

然后先给出判断：

> 这可能是绝对路径泄漏或路由前缀问题，先检查返回的 share path 和 Caddy `/share/*` 兼容路由。

这才叫复用。

### 4. 每次复用后更新经验

经验不是写完就死了。

如果后续发现：

- 新版本改了路径规则
- 旧判断不再成立
- 又出现新的边界条件

就应该更新经验卡。

这和工程文档一样：过期的经验比没有经验更危险。

## 十、经验压缩的质量标准

不是所有总结都叫经验。

我会用这几个标准判断它是否值得进入知识库：

### 1. 可触发

它要写清楚什么时候该想起这条经验。

差：

```text
Caddy 配置问题总结
```

好：

```text
当应用从根路径迁到 /xxx/ 后，分享链接或静态资源跳回根路径时
```

### 2. 可诊断

它要写清楚症状和根因之间的关系。

### 3. 可执行

它要有操作步骤，不只是感想。

### 4. 可验证

它要说明怎么确认修好了。

### 5. 可迁移

它不能只绑定一个命令，而要抽象出结构。

例如：

```text
检查绝对路径泄漏
```

比：

```text
把 /share 改成 ./share
```

更有迁移性。

## 十一、对“大模型训练”的启发

如果从大模型训练角度看，这里有一个很重要的分层。

### 上下文学习

模型在当前 prompt 里看到例子，然后临时遵循。

这很像工作记忆。

### RAG

模型从外部知识库拿材料进上下文。

这很像翻资料。

### 经验卡片

人或系统把过去会话压缩成可复用结构，再让模型检索。

这很像外部化的情景记忆和程序性提示。

### 微调或继续训练

把模式写进模型参数。

这才更接近“模型本体学会了”。

但对个人开发者和日常工具来说，微调不是最优先的。更实际的路线是：

```text
会话 -> 经验卡片 -> 检索 -> 上下文注入 -> 执行 -> 更新经验
```

这条路线成本低、可解释、可审计，也更适合项目知识。

---

## 十二、我接下来想怎么做

对我自己的工作流来说，我希望每次比较有价值的 Codex / Claude Code 会话，都能在结束时多一个动作：

```text
是否沉淀为经验卡片？
```

如果是，就生成：

- 一份项目内经验卡
- 一段可放进博客的复盘
- 一条未来 Agent 可以检索的触发条件

这样博客不只是展示内容，知识库也不只是存档。

它们会形成一个循环：

```text
做事 -> 复盘 -> 压缩 -> 检索 -> 复用 -> 再做事
```

这就是我理解的“知识利用流动”。

不是知识被收藏起来，而是知识在下一次行动里继续发生作用。

---

## 结论

单次感知能力和经验压缩确实有内在相似性：它们都说明一次关键经验可以改变后续识别。

但工程经验压缩更进一步，它不只让你“看见”，还要让你“能处理”。

对今天的 AI 工具来说，真正的问题不是能不能把所有会话塞进知识库，而是能不能把一次会话压缩成结构化、可触发、可验证、可迁移的经验对象。

RAG 能帮我们找回经验。

但经验本身，需要被认真写好。
